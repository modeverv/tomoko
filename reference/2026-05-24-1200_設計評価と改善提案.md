```text
recent session context is baseline
```

```python
async def build_context(input_text: str, policy: ContextBuildPolicy) -> ContextSnapshot:
    deadline = time.monotonic() + policy.max_build_ms / 1000
    trace = ContextBuildTrace()

    snapshot = ContextSnapshot()

    async def remaining_ms() -> float:
        return max(0, (deadline - time.monotonic()) * 1000)

    # 1. 最低限: 同一セッション
    snapshot.same_session_turns = await with_budget(
        read_same_session_turns(),
        timeout_ms=remaining_ms(),
        trace=trace,
        stage="same_session",
        required=True,
    )

    # 2. 余裕があれば最近の会話
    if remaining_ms() > policy.min_stage_ms:
        snapshot.recent_turns = await with_budget(
            read_recent_turns(),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="recent_turns",
        )

    # 3. 余裕があれば長期記憶
    if remaining_ms() > policy.min_stage_ms:
        snapshot.session_memory_hits = await with_budget(
            search_session_summaries(input_text),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="session_summary_search",
        )

    # 4. さらに余裕があれば細かい turn memory
    if remaining_ms() > policy.min_stage_ms:
        snapshot.turn_memory_hits = await with_budget(
            search_turn_embeddings(input_text),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="turn_memory_search",
        )

    snapshot.trace = trace
    return snapshot
```

ここすら取れないなら DB か設計が壊れている可能性が高い。

**Tomoko の賢さと応答速度のトレードオフを、設定で操作できる**

```python
        "turn_memory": asyncio.create_task(search_turn_embeddings(input_text)),
        "persona": asyncio.create_task(read_persona_state()),
        "lexicon": asyncio.create_task(read_latest_lexicon()),
    }

    done, pending = await asyncio.wait(
        tasks.values(),
        timeout=policy.max_build_ms / 1000,
    )

    snapshot = ContextSnapshot()

    for name, task in tasks.items():
        if task in done and not task.cancelled() and task.exception() is None:
            snapshot.add(name, task.result())
            trace.mark_included(name)
        else:
            task.cancel()
            trace.mark_timeout(name)

    snapshot.apply_priority_and_token_budget(policy)
    snapshot.trace = trace
    return snapshot
```

この設計の良いところは、**context 取得方法を増やしても、応答前の最大待ち時間が増えない**ことです。

普通は機能を足すと、

```text
記憶を増やす
→ 検索が増える
→ context 生成が遅くなる
→ 応答が遅くなる
```

となる。

でも並列 DB I/O + timeout + priority にすると、

```text
記憶取得源を増やす
→ 同時に投げる
→ 時間内に返ったものだけ使う
→ 遅いものは trace に残して改善対象にする
```

になる。

これは長期運用にかなり強いです。

ただし注意点もあります。

一つ目は、**connection pool を食い潰さないこと**です。  
1応答につき 6〜8 クエリを並列に投げるなら、同時会話数が増えたときに pool が詰まります。家庭内 Tomoko なら当面大丈夫だと思いますが、設計上は、

```text

db_pool_size = 10 or 20
```

みたいに明示しておいた方がよいです。

二つ目は、**重い vector search を無制限に並列投入しないこと**です。  
session summary 検索と turn embedding 検索を毎回両方走らせると、ログが増えたときに重くなる可能性があります。

なので初期は、

```text
常に走らせる:
  same_session
  recent_turns
  session_summary_search

条件付き:
  turn_embedding_search
  persona / lexicon / diary
```

くらいでもよいと思います。

三つ目は、**返ってきた順に雑に詰めないこと**です。  
並列にすると早く返ったものが勝ちやすくなりますが、context の価値は速度だけでは決まりません。

なので最後に必ず、

```text
priority
recency
same_session 優先
relevance
token budget
重複除去
```

で整形する必要があります。

つまり、構造としてはこうです。

```text
ContextSnapshotBuilder
  ├─ ParallelContextFetcher
  │    ├─ same_session query
  │    ├─ recent_turns query
  │    ├─ session_summary query
  │    ├─ turn_memory query
  │    ├─ persona query
  │    └─ lexicon query
  │
  ├─ ContextTrace
  │    ├─ timing
  │    ├─ timeout
  │    ├─ included/skipped
  │    └─ candidate_count
  │
  └─ ContextAssembler
       ├─ priority
       ├─ dedup
       ├─ token budget
       └─ final prompt context
```

結論として、これはかなり良い判断だと思います。

**DB を唯一の真実にするなら、context 生成は DB に対する時間制限付き多重 I/O として設計する。**

これは自然です。

しかも、長期運用時に問題が出たとき、

```text
DB性能の問題
クエリ設計の問題
索引設計の問題
取得戦略の問題
context選別の問題
LLM側の問題
```

を分離して見られる。

この分離ができる時点で、かなり「長期運用するつもりの設計」になっています。  
まさに、今やろうとしている Tomoko の規模感なら、ここに抽象を挟む価値は十分あります。

しかもこれは「難しい概念を使っている」ではなく、**難しいもの同士が同時に干渉する**タイプの難しさですね。

要するに Tomoko は、

```text
ノンブロッキング
  応答生成・STT・TTS・DB検索・再生制御で詰まらない

  attention / playback / speaking / session / turn status が破綻しない
```

を全部同時に満たさないといけない。

これ、普通に難しいです。

特に厄介なのは、**非同期処理は順番を崩すが、状態機械は順番に依存する**ところです。

たとえば、

```text
1. Tomoko が話し始める
2. audio_start を送る
3. バイナリ音声 chunk を送る
4. クライアントが playback_started を返す
5. 人間の割り込み音声が入る
6. STT が遅れて transcript を返す
7. playback_ended が返る
8. BargeInDetector が分類する
```

この順序が毎回きれいに揃うとは限らない。  
でも状態としては、

```text
今の transcript は、
Tomoko の再生中の回り込みなのか？
再生終了直後の echo grace なのか？
本物の割り込みなのか？
次の会話なのか？
```

を判断しないといけない。

ここがかなり難所です。

```text
WebSocket message
Audio chunk
STT result
Playback telemetry
LLM token
TTS chunk
Timeout
```

イメージとしては、

```text
外界イベント
  ↓
Event Queue
  ↓
TomoroSession reducer
  ↓
状態更新
  ↓
副作用コマンド生成
    - DB write
    - LLM start
    - TTS start
    - audio_control stop
    - websocket send
```

みたいにしたい。

React/Redux 的に言えば reducer、ゲームサーバー的に言えば tick/update loop、分散システム的に言えば event-sourced state machine に近いです。

```python
        self.is_speaking = False
        await self.db.write(...)
        await self.start_new_reply(...)
```

```python
def reduce(state, event) -> tuple[new_state, list[Command]]:
    ...
```

```text
状態更新は同期的・短時間・atomic に近く
重い処理は Command として外に出す
Command の結果はまた Event として戻す
```

```text
deadline
timeout
cancellation
idempotency
turn_id
chunk_id
session_id
state transition log
```

が必要になる。

特に `turn_id` / `chunk_id` を入れているのはかなり正しいです。  
非同期では「古い結果が遅れて返ってくる」ことが普通にあるので、

```text
この TTS chunk は今の turn のものか？
```

を判定できないと破綻します。
