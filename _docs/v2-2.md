# v2-2: 畳み掛け発話（floor-holding consecutive utterances）構想と実装計画

作成: 2026-06-12 ／ 前提: PLAN.md の Phase TT-v2.10d（fusion トリガー切替）完了後に着手する。

## 0. 背景

Neuro-sama（Vedal 作の AI VTuber）のデモに、PC 画面を見ながら

> 「ねぇ、あなたはさっきからコピペばっかりしているわね」
> 「コピペするのが好きなの？」
> 「コピペするのがプログラミングなの？」

と**畳み掛けるように連続発話**する場面がある。煽り口調はエンタメ要素（ペルソナ側の問題）なので対象外とし、本ドキュメントは「畳み掛け」という**会話構造**を tomoko で実現するための分解と計画を扱う。

## 1. 「畳み掛け」の技術的分解

畳み掛けは5つの部品に分解できる:

| # | 部品 | 内容 |
|---|------|------|
| 1 | 刺激源 | 画面を見て「コピペばかりしている」という観測を得る |
| 2 | 発話候補の生成 | 観測から「言いたいこと」を作る |
| 3 | 自発発話のトリガー | ユーザーが話していない良いタイミングで切り出す |
| 4 | **床（floor）の保持** | 一発話で終わらず、ユーザーが床を取りに来なければ続ける |
| 5 | **エスカレーション連鎖** | 「無反応だった」ことを文脈に次の一手を重ねる |

このうち 1〜3 は tomoko に骨格が存在し、**本質的に新規なのは 4 と 5 のみ**。

## 2. 現状資産マッピング

| 部品 | tomoko の現状 | 状態 |
|------|--------------|------|
| 刺激源（画面） | `server/world_observations/`（markdown ingest → normalizer → interpreter）。ただしスクショ→視覚理解の入口は無い。`GemmaMLXBackend` が mlx_vlm ベース = **視覚対応モデルは既にスタックに居る** | 🔶 コレクタのみ新規 |
| 発話候補の生成 | `candidate_gen`（backend-trace に実績 1300件超）+ thinker | ✅ |
| 自発発話トリガー | `server/gateway/initiative_policy.py`（DesireLoadAverages / SpeakabilityLoadAverages / CandidateSpeakPolicy / InitiativeLLMJudge） | 🔶 floor 信号との結線が未実装 |
| 床の状態の把握 | turn-taking v2 レーン: VAD 状態・audio_level・p_yielding（VAP）・fusion_score。TT-v2.10 で advisory/ログに揃った | ✅ |
| 割り込まれたら即中断 | 仮推論の破棄ライフサイクル（TT-v2.4）と同型の機構が必要 | 🔶 発話中断（barge-in）への転用が必要 |
| 床保持ループ | **存在しない**。現在の会話モデルは厳密な交互ターン（一度話したら必ずユーザーの番を待つ） | ❌ 新規 |
| 追撃候補の先読み | TT-v2.10c の prefill/投機機構が転用可能（自分の発話再生中、LLM は暇） | 🔶 転用 |

## 3. 設計原則（既存規約の継承）

- **一方向データフロー**を維持する。画面コレクタは `world_observations` への新しい書き手であり、逆流は作らない。
- **床保持ポリシーは session 内に閉じる**。「交互ターン前提」を崩す唯一の箇所なので、状態機械として明示的に書く（暗黙のフラグ分岐にしない）。
- **shadow/initiative レーンは助言のみ**。実際に発話する決定（actuator）は今まで通り session に一本化。
- **log-only → 実機ログ突合 → 切替**の段階導入を踏襲する。
- 新しい挙動はまず **shadow_bench でシミュレート**し、人間の操作なしで傾向を出してから実装する（TT-v2.8/2.9 の方法論）。

## 4. 床保持の状態機械（中核設計）

現在: `発話完了 → listening（ユーザーの番）` の固定遷移。

提案: 発話完了後に **HOLDING** 状態を挟む。

```
tomoko 発話完了
   │
   ▼
┌─────────── HOLDING（床保持・最大 N 回 / 最長 T 秒） ───────────┐
│  沈黙タイマー起動（pause_ms: 800〜1500ms、ベンチで決定）        │
│                                                                │
│  ユーザー発話開始（VAD listening / audio_level 上昇）          │
│      → 即座に床を譲る（追撃候補を破棄）→ LISTENING            │
│                                                                │
│  pause_ms 経過 かつ 追撃候補あり かつ desire 高                │
│      → 追撃発話を再生 → HOLDING を継続（カウンタ+1）          │
│                                                                │
│  pause_ms 経過 かつ 追撃候補なし（or desire 減衰 or 上限到達） │
│      → 床を譲る → LISTENING                                    │
└────────────────────────────────────────────────────────────────┘
```

判定には既存の motivation 評価と同じ形の加重スコアを使う（fusion の設計の鏡像）:

```
hold_score = w_desire × 追撃desire（候補の鮮度・話題圧）
           + w_floor  × 床の空き（ユーザー沈黙の継続）
           − w_fatigue × 連続発話回数ペナルティ
続行 = hold_score >= θ_hold
```

重みは shadow_bench 拡張（TT-v3.0）でグリッドサーチして決める。

## 5. 追撃候補の先読み（エスカレーション連鎖）

自分の発話 i を TTS 再生している間（LLM はアイドル）、次を投機生成する:

- 入力: 発話 i までの自分の発話列 + 「ユーザーは無反応だった」という条件
- 出力: 発話 i+1 の候補（短い一手。proposal 階層の backchannel / short_confirmation 相当の長さ）
- ユーザーが反応したら破棄（仮推論 discard と同じライフサイクル）
- prefill 済みコンテキストを使うため、発話間ポーズ内（~1秒）に生成が間に合う

レイテンシー的にはタダ同然: 生成は発話再生の影で終わっており、HOLDING のポーズは**意図的な間**（タイミング芸）であって計算待ちではない。

## 6. フェーズ分割

### Phase TT-v3.0: shadow_bench 畳み掛けシナリオ（シミュレーション先行）

実装前に、発話間ポーズ長と hold_score 重みの傾向をベンチで出す。

- [ ] `scripts/shadow_bench.py` に「tomoko 発話後のユーザー反応」シナリオを追加する: say で tomoko 発話を模した音声を作り、(a) ユーザーが pause 内に話し始めるケース（シードに `reaction` ラベル群を追加）、(b) 無反応ケースをシミュレート。
- [ ] HOLDING 状態機械を純関数として `scripts/` 内に実装し（後で本実装の参照になる）、pause_ms ∈ {600, 800, 1000, 1200, 1500} × hold 重みのグリッドで「誤続行率（ユーザーが話し始めたのに被せた）」「譲りすぎ率（無反応なのに黙った）」を計測する。
- [ ] 分析レポート + HTML に畳み掛けタイムライン（発話1→pause→発話2→…、ユーザー反応マーカー）を追加する。

### Phase TT-v3.1: 床保持ポリシー（session 実装）

- [ ] `server/session.py` に HOLDING 状態を追加する（VadState とは独立した会話床の状態。`_floor_state: "listening" | "speaking" | "holding"` など明示的に）。
- [ ] tomoko の TTS 再生完了イベントで HOLDING に遷移し、ベンチで決めた pause_ms の沈黙タイマーを起動する。
- [ ] タイマー満了時: 追撃候補キューが空でなく hold_score ≥ θ なら次発話を再生、それ以外は LISTENING へ。連続発話は最大 N 回（初期値 3）・合計 T 秒（初期値 15s）でハードキャップ。
- [ ] ユーザー発話開始（VAD listening 遷移 or audio_level 閾値超え）で即座に: 再生中の TTS を停止（barge-in）、追撃キュー破棄、LISTENING へ。停止イベントを `logs/turn-taking-main.jsonl` に記録。
- [ ] **log-only 段階**: 最初は「続行していたはず」のイベントを記録するだけにして、実会話ログで誤続行率を確認してから実発話を有効化する。
- [ ] ユニットテスト: 状態遷移（満了→続行 / ユーザー割り込み→譲る / 上限到達→譲る / 候補なし→譲る）。

### Phase TT-v3.2: 追撃候補の先読み生成

- [ ] 自分の発話再生開始時に、`「直前の自分の発話列 + ユーザー無反応」` を条件にした追撃候補生成を背景タスクで起動する（E2B 級の速いモデル、`trace_role="follow_up_gen"`）。
- [ ] 生成結果を追撃キュー（ターン内 append-only、床を譲ったら破棄）に積む。
- [ ] TT-v2.10c の prefill API を使い、自分の発話中にコンテキストを温めておく。
- [ ] ユニットテスト: 破棄ライフサイクル（ユーザー反応で discard）、キュー上限。

### Phase TT-v3.3: 画面コレクタ（刺激源の追加）

- [ ] `screencapture`（macOS）で定期スクショ → `GemmaMLXBackend`（mlx_vlm）で一行描写 → `world_observations` の ingest 形式（markdown）に書き出すコレクタを `server/world_observations/` 配下に追加する。
- [ ] 描写の差分検出（前回と同じ内容なら ingest しない）で観測ノイズを抑える。
- [ ] interpreter → candidate_gen → initiative_policy の既存経路で「画面についての発話候補」が生まれることをログで確認する。
- [ ] プライバシー規約: スクショ画像そのものは保存せず、一行描写テキストのみを残す。コレクタは明示的な opt-in（env フラグ）。

### Phase TT-v3.4: initiative レーンとの結線（自発発話の床判定）

- [ ] v2 レーンの床信号（quiet / p_yielding / VAD 状態の集約値）を initiative_policy の speakability 入力に渡す（advisory → policy の一方向追加配線）。
- [ ] 「ユーザーが黙っていて床が空いている + 画面観測由来の候補がある」場合に自発発話 → そのまま TT-v3.1 の HOLDING に入れば、観測→自発→畳み掛けの全連鎖が完成する。

### Phase TT-v3.5: 服従/欲求の裁定（bounded disobedience）

「黙って」と言われたとき、指示に従うか自分の発話欲を優先するかを**状況によって裁定する**仕組み。
ルールで黙る存在は道具だが、裁定で黙る存在は黙り方に分散が出る — その分散のパターンが
外から観測される「人格」になる。

#### 現状資産

- `server/gateway/stop_intent.py` + session の `stop_intent_classified` イベント処理が既にあり、
  「黙って」系発話の**認識**は存在する。ただし現在は無条件ルール（stop と分類されたら従う）。
- `server/gateway/initiative_policy.py` に `PersonalityDynamics`（人格係数）と
  EMA ベースの LoadAverages（時間減衰する圧力）が既にあり、裁定の置き場は用意済み。

#### 設計: ルールから裁定への格上げ

fusion（応答タイミング）・hold_score（床保持）・initiative（自発）と同じ加重スコア競合の
4つ目のインスタンスとして実装する:

```
obey_score   = w_command × 指示の強さ（明示度・口調） × 累積圧（compliance pressure）
desire_score = w_desire  × 言いたいことの重要度・鮮度 × 人格係数（素直さ/反骨）

黙る = obey_score > desire_score
```

#### 絶対規則: 不従順は有界（bounded）であること

この仕組みを動的で魅力的にしつつ壊れた存在にしないための構造的保証:

1. **累積圧の単調性**: 「黙って」が繰り返される/語気が強まるたびに compliance pressure が
   EMA で単調増加し、**数回以内に必ず obey 側が勝つ**ことを重み設計で保証する
   （desire_score の上限 < 累積後の obey_score 下限、をユニットテストで固定する）。
2. **初回の余地**: desire が十分高い場合のみ、1回目の「黙って」に対して「一手だけ通す」
   ことがありうる。2回目以降は必ず従う。
3. **記憶と残効**: 従わなかった事実を記録し、以後しばらく compliance 重みを上げたままにする
   （「さっき怒られたから今は大人しい」という残効が、観測可能な性格の連続性になる）。
4. **最終制御権は常に人間側**: 明示的な停止コマンド（UI/設定経由）は裁定を経由せず
   無条件に従う。裁定の対象は会話内の自然言語指示のみ。

#### 実装ステップ

- [ ] stop_intent 分類に「強さ」推定を追加する（明示度: 「ちょっと静かに」<「黙って」<「黙れ」、
      繰り返し回数、直前の不従順の有無）。
- [ ] compliance pressure を EMA で管理する状態を session に追加（ターンを跨いで減衰、
      stop 指示で増加、不従順でさらに増加）。
- [ ] TT-v3.1 の HOLDING / barge-in 処理に裁定を組み込む: ユーザー発話が stop 指示と
      分類された場合のみ「無条件中断」ではなく obey/desire 裁定を通す。
      stop 以外の通常発話による barge-in は従来通り無条件で床を譲る。
- [ ] 裁定結果（obey_score / desire_score / 判定 / 累積圧）を `logs/turn-taking-main.jsonl` に
      記録する（log-only で分散の観測から始める）。
- [ ] ユニットテスト: 有界性（指示2回目で必ず従う）、残効（不従順後の compliance 上昇）、
      明示停止コマンドの無条件性。
- [ ] `PersonalityDynamics` に素直さ/反骨パラメータを追加し、w_desire の人格係数として配線する。

#### 検証

shadow_bench に「stop 指示への反応」シナリオを追加し（シードに `stop` ラベル群:
「黙って」「ちょっと静かにして」「うるさい」等）、人格係数×累積圧のグリッドで
「2回目で従う保証」が崩れないことをシミュレーションで確認してから実装する。

## 7. 検証方法

1. **ベンチ先行**（TT-v3.0）: 誤続行率・譲りすぎ率・体感テンポ（発話間隔分布）を人間なしで計測。
2. **log-only**（TT-v3.1）: 実会話で「続行していたはず」イベントとユーザーの実際の行動を突合。誤続行率がベンチ予測と整合したら有効化。
3. **有効化後**: `analyze_turn_taking_v2.py` 系に畳み掛けセッションの分析（連続発話回数分布、barge-in 中断までの反応時間）を追加。

## 8. リスクと対策

| リスク | 対策 |
|--------|------|
| 被せ（ユーザーが話し始めたのに続行） | barge-in 即中断 + log-only 検証で誤続行率を先に確認 |
| しつこさ（畳み掛けすぎ） | 回数 N・時間 T のハードキャップ + fatigue ペナルティ + ペルソナ側で抑制 |
| 交互ターン前提の他コンポーネントとの不整合 | 床状態を明示的な状態機械にし、HOLDING 中の partial 処理・shadow レーンの挙動をユニットテストで固定 |
| 画面観測のプライバシー | 描写テキストのみ保存・opt-in・差分検出で最小化 |
| 不従順の暴走（黙れと言っても黙らない） | TT-v3.5 の有界性保証（累積圧の単調性 + 2回目で必ず従う重み設計をテストで固定 + 明示停止コマンドは裁定を経由しない） |
