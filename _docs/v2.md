# Turn-taking v2 / Semantic Speech Decision Shadow Lane 設計メモ

## 目的

Tomoko の turn-taking を、現行の `VAD / final transcript` 主導から、将来的に `partial STT / semantic saturation / speech motivation` を含む計算モデルへ拡張する。

ただし、これは会話制御の本質的変更であり、いきなり main route に入れると runtime behavior、reply routing、TTS ordering、DB ordering、conversation lifecycle が複雑化し、hot path を壊す危険が高い。

そのため、最初は **turn-taking v2 shadow lane** として、main route とは分離した観測・分析用の時間軸を作る。

v2 は最初、発話制御に一切関与しない。

---

## 背景となる設計思想

Tomoko の基本思想は以下である。

```text
hot path を死守する
余計な複雑性を main route に入れない
LLM に制御フローを渡さない
TomoroSession を final owner にする
```

TomoroSession は会話状態の final owner であり、外部 worker / LLM / DB / TTS / playback などの結果は input として戻す。ただし、最終的な state transition / reply start / lifecycle / gate 判断は TomoroSession が所有する。

今回の turn-taking v2 は、最終的には main route に関わる可能性があるが、最初から制御に混ぜてはいけない。

---

## 現行 turn-taking v1 の位置づけ

現行 v1 は概ね以下の責務である。

```text
final transcript が来る
  -> rule-first TurnTakingJudge
  -> noise / backchannel / stop / follow-up / new input を分類
  -> 曖昧な場合だけ worker 補助
  -> TomoroSession が最終制御
```

v1 は保守対象として残す。

v1 に v2 の概念を無理に足すと責務が混ざる。

v2 は別の目的を持つ。

---

## turn-taking v2 の目的

v2 は、final transcript 後の分類器ではなく、partial STT の時間軸を使って以下を推定する。

```text
partial STT stream
  -> hallucination / noise check
  -> stable prefix extraction
  -> semantic saturation
  -> remaining information risk
  -> semantic split risk
  -> speech motivation
  -> speech decision score
  -> provisional inference / prepare-only の妥当性
```

v2 の本質は、以下である。

```text
「音が終わったか」ではなく、
「意味的にもう聴き終わったか」
「Tomoko がどれくらい話したいか」
「今入ると邪魔か」
を連続値として扱う。
```

---

## 計算モデルの基本案

### SpeechDecisionScore

発話決意を 1 つの数値として扱う。

```text
SpeechDecisionScore S(t) ∈ [0, 1]
```

概念式:

```text
S(t) = HardGate(t) × sigmoid(
    + wD * Desire(t)
    + wU * Understanding(t)
    + wM * MaAI_Affordance(t)
    + wC * ContextRelevance(t)
    + wR * RelationshipPermission(t)
    - wI * InterruptionRisk(t)
    - wV * VadPenalty(t)
    - wP * PlaybackPenalty(t)
)
```

重要点:

```text
VAD は主役ではない。
VAD は penalty の一要素に下げる。
```

### 出力段階

```text
S < 0.25
  silence

0.25 <= S < 0.45
  prepare only

0.45 <= S < 0.60
  backchannel

0.60 <= S < 0.75
  short confirmation

0.75 <= S < 0.90
  full response candidate

0.90 <= S
  floor grab candidate
```

ただし、最初の実装では v2 は一切発話させない。

## Semantic Finish / Listening Completion

partial transcript の更新ごとに、ユーザー発話の意味的完了度を推定する。

```text
SemanticFinishState:
  semantic_saturation
  remaining_info_risk
  semantic_split_risk
  safe_response_level
  confidence
```

例:

```text
partial: 「昨日の件なんだけど」
  -> semantic_saturation low
  -> remaining_info_risk high

partial: 「昨日の件なんだけど、進めていいと思う？」
  -> semantic_saturation high
  -> remaining_info_risk low

partial: 「昨日の件なんだけど、進めていいと思う？ ただ」
  -> semantic_split_risk high
  -> provisional result should become stale
```

日本語では「ただ」「でも」「というか」「前提として」「一個だけ」などの後置条件により、意味が後から分裂する。このため `semantic_split_risk` を明示的に持つ。

---

## Speech Motivation

Tomoko 側の発話モチベーションは、LLM の都度推論ではなく、状態変数として保持する。

```text
SpeechMotivationState:
  desire_short
  desire_mid
  desire_long
  unspoken_pressure
  candidate_pressure
  curiosity_pressure
  reply_prepared_pressure
  recent_rejection_penalty
  last_updated_at
```

時間経過では指数減衰する。

```text
desire_short *= exp(-dt / tau_short)
desire_mid   *= exp(-dt / tau_mid)
desire_long  *= exp(-dt / tau_long)
```

イベントで加算・減算する。

```text
candidate generated
  -> candidate_pressure += priority * urgency

semantic saturation increased
  -> reply_prepared_pressure += semantic_saturation

provisional reply generated
  -> reply_prepared_pressure += confidence

semantic split detected
  -> reply_prepared_pressure -= semantic_split_risk

user says "ちゃんと聞いて"
  -> recent_rejection_penalty += penalty
  -> desire_short -= penalty
```

これにより、音声入力がない状態でも自然発話候補を同じモデルで扱える。

## partial STT の扱い

v2 の前提として、partial STT の時間軸を session 内で扱う必要がある。

しかし、partial transcript は不安定であり、hallucination や揺れを含むため、そのまま信じない。

必要な前処理:

```text
PartialTranscriptStream
  -> TranscriptValidity
  -> StablePrefixExtractor
  -> SemanticFinishJudge
```

### TranscriptValidity

軽量ルールまたは軽量推論で、意味のある STT かどうかを判定する。

落とす候補:

```text
短すぎる
定型 hallucination
前 partial から急に意味が飛ぶ
音量が低すぎる
同じ文字列の反復
言語が突然変わる
```

### StablePrefixExtractor

partial STT は揺れるため、安定して伸びている prefix を重視する。

```text
前回 partial:
  昨日の件なんだけど、あれ

今回 partial:
  昨日の件なんだけど、あれ進めていいと思う

stable prefix:
  昨日の件なんだけど、あれ
```

stable prefix が伸びているときだけ、semantic saturation の信頼度を上げる。

## Provisional Inference

v2 は、semantic saturation が高い場合に「今なら推論開始してよさそう」という advisory を出せる。

ただし、最初は実際の LLM 推論も発話も行わない。

将来段階では、v2 は provisional inference を作る可能性がある。

```text
ProvisionalReply:
  based_on_observation_id
  based_on_stable_text
  based_on_intent_hash
  created_at
  response_text
  status: pending / valid / stale / discarded
```

新しい partial / final transcript が来たら照合する。

```text
if new intent diverges:
  discard provisional reply

if semantic_split_risk rises:
  discard or downgrade

if final transcript confirms intent:
  promote to speakable candidate
```

重要:

```text
推論すること != 発話すること
```

v2 が推論したとしても、TomoroSession final gate が通すまで発話しない。

---

## なぜ v2 shadow lane が必要か

この機構を main route に直接入れると、以下が混ざる。

```text
なぜ喋ったのか
なぜ待ったのか
なぜ推論結果を捨てたのか
v1 と v2 のどちらが決めたのか
reply_done はどちらの責務か
TTS flush はどうなるのか
DB write ordering は変わったのか
conversation lifecycle と競合しないか
```

これは hot path に対して危険である。

そのため、v2 はまず shadow lane として分離する。

---

## v2 shadow lane の最初の目的

最初の目的は、v2 の正しさではない。

目的は以下である。

```text
partial STT observation が DB に乗る
v2 worker が低遅延で拾える
v2 advisory が DB に戻る
main 側が advisory を受け取れる
main は advisory をログに吐くだけ
runtime behavior は変わらない
```

これは「設計」以前の実現可能性 scaffold である。

---

## PostgreSQL LISTEN/NOTIFY の限定利用

現行方針では pub/sub は原則禁止であり、PostgreSQL が唯一の真実である。

今回も汎用 pub/sub は解禁しない。

ただし、polling だけでは v2 が partial STT を現実的な低遅延で拾えない可能性がある。

そのため、PostgreSQL LISTEN/NOTIFY を **non-authoritative wakeup notification** として限定利用する案を採る。

### 原則

```text
LISTEN/NOTIFY は状態を運ばない
payload は id のみ
source of truth は必ず DB row
main route は v2 の応答を待たない
v2 が落ちても main route は壊れない
worker は recovery polling を持つ
```

これは pub/sub 解禁ではなく、DB-backed wakeup notification である。

---

## 最小ループ

```text
partial STT
  -> partial_transcript_observations INSERT
  -> pg_notify("turn_taking_v2_observation", observation_id)

turn-taking-v2 worker
  -> LISTEN turn_taking_v2_observation
  -> observation_id で DB row を読む
  -> advisory を生成
  -> turn_taking_v2_advisories INSERT
  -> pg_notify("turn_taking_v2_advisory", advisory_id)

main-side listener
  -> LISTEN turn_taking_v2_advisory
  -> advisory_id で DB row を読む
  -> server-debug.log / JSONL に出す
  -> 制御には使わない
```

main-side listener は制御入力ではない。

```text
受け取る
読む
ログに出す
捨てる
```

だけである。

## 分析ツール案

v2 を main route に混ぜる前に、会話後の突合ツールを作る。

```bash
python -m tools.analyze_turn_taking_v2 \
  --session-id <conversation_session_id> \
  --main logs/turn-taking-main.jsonl \
  --v2 logs/turn-taking-v2-shadow.jsonl \
  --out reports/turn-taking/<session-id>.md
```

出力例:

```markdown
# Turn-taking v2 analysis

## Summary

- v2 proposals: 18
- would_start_inference: 5
- valid early prepares: 3
- stale/discard needed: 1
- dangerous early proposals: 1
- average lead time: 420ms

## Timeline

| time | main | v2 shadow | outcome |
|---|---|---|---|
| +1.20s | partial: 昨日の件... | score 0.42 wait | safe_wait |
| +1.86s | partial: 進めていいと思う | score 0.76 prepare | good_early_prepare |
| +2.31s | final transcript | main start_reply | v2 was 450ms early |
```

---

## 評価分類

```text
good_early_prepare
  v2 が早めに推論へ行こうとし、final transcript と整合した

too_early_wrong
  v2 が早すぎて、後続 partial / final で意味が変わった

missed_opportunity
  v2 が黙っていたが、人間的には早く準備できた

safe_wait
  v2 が待った判断は妥当だった

dangerous_speak
  v2 score が高かったが、実際は「ちゃんと聞けよ」系だった
```

この分析により、v2 が「なんとなく良さそう」ではなく、何 ms 早く、何件正しく、何件危険だったかを評価できる。

## 実装フェーズ案

### Phase TT-v2.0: feasibility scaffold

目的:

```text
DB-backed wakeup notification が partial STT observation の shadow path として成立するか確認する。
```

やること:

```text
partial_transcript_observations を append する
observation_id だけを NOTIFY する
v2 worker が LISTEN して DB row を読む
dummy advisory を append する
advisory_id だけを NOTIFY する
main-side listener が advisory を読んでログに吐く
```

やらないこと:

```text
発話制御
reply start
TTS
audio hot path
reply_done routing
DB write ordering 変更
conversation lifecycle 変更
candidate final gate 変更
```

### Phase TT-v2.1: shadow advisory

```text
hallucination check
stable prefix extraction
semantic saturation
speech decision score
safe_response_level
```

を v2 worker が出す。

まだ制御には使わない。

### Phase TT-v2.2: analysis tool

main と v2 の timeline を突合し、Markdown report を生成する。

### Phase TT-v2.3: prepare-only dry run

v2 が「今なら推論に行きたい」を出す。

ただし実際の LLM 推論はまだ行わないか、dry-run として記録するだけ。

### Phase TT-v2.4: provisional inference

v2 が低頻度で provisional inference を作る。

ただし発話しない。

新しい partial / final transcript で stale discard が正しく働くかを確認する。

### Phase TT-v2.5 以降

人間が分析 report を見て納得した場合のみ、以下を検討する。

```text
backchannel only
short confirmation
full response candidate
```

いずれも TomoroSession final gate を通す。

---

## 絶対に守る条件

```text
v2 は最初 shadow-only
LISTEN/NOTIFY payload は id のみ
DB row が source of truth
main は v2 の応答を待たない
main は advisory をログするだけ
v2 の失敗は main に影響しない
recovery polling を持つ
発話制御・reply routing・TTS・audio hot path・DB ordering・lifecycle は変えない
人間が timeline report を見て納得するまで gate に混ぜない
```

---

## 結論

turn-taking v2 は、現行 v1 の小改良ではなく、会話制御モデルの本質的な拡張である。

そのため、main route に直接入れるべきではない。

最初は、DB-backed wakeup notification と shadow advisory log により、partial STT の時間軸と v2 の判断時間軸が技術的に成立することを確認する。

この scaffold は複雑性の導入ではあるが、main route を守るための最小限の複雑性である。

v2 は「喋らせる仕組み」ではなく、まず「観測し、数値化し、後で人間が突合する仕組み」として始める。
