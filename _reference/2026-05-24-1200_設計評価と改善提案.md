recent session context is baseline

```python
async def build_context(input_text: str, policy: ContextBuildPolicy) -> ContextSnapshot:
    deadline = time.monotonic() + policy.max_build_ms / 1000
    trace = ContextBuildTrace()

    snapshot = ContextSnapshot()

    async def remaining_ms() -> float:
        return max(0, (deadline - time.monotonic()) * 1000)

    # 1. 最低限: 同一セッション
    snapshot.same_session_turns = await with_budget(
        read_same_session_turns(),
        timeout_ms=remaining_ms(),
        trace=trace,
        stage="same_session",
        required=True,
    )

    # 2. 余裕があれば最近の会話
    if remaining_ms() > policy.min_stage_ms:
        snapshot.recent_turns = await with_budget(
            read_recent_turns(),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="recent_turns",
        )

    # 3. 余裕があれば長期記憶
    if remaining_ms() > policy.min_stage_ms:
        snapshot.session_memory_hits = await with_budget(
            search_session_summaries(input_text),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="session_summary_search",
        )

    # 4. さらに余裕があれば細かい turn memory
    if remaining_ms() > policy.min_stage_ms:
        snapshot.turn_memory_hits = await with_budget(
            search_turn_embeddings(input_text),
            timeout_ms=remaining_ms(),
            trace=trace,
            stage="turn_memory_search",
        )

    snapshot.trace = trace
    return snapshot
```

ここすら取れないなら DB か設計が壊れている可能性が高い。

**Tomoko の賢さと応答速度のトレードオフを、設定で操作できる**

        "turn_memory": asyncio.create_task(search_turn_embeddings(input_text)),
        "persona": asyncio.create_task(read_persona_state()),
        "lexicon": asyncio.create_task(read_latest_lexicon()),
    }

    done, pending = await asyncio.wait(
        tasks.values(),
        timeout=policy.max_build_ms / 1000,
    )

    snapshot = ContextSnapshot()

    for name, task in tasks.items():
        if task in done and not task.cancelled() and task.exception() is None:
            snapshot.add(name, task.result())
            trace.mark_included(name)
        else:
            task.cancel()
            trace.mark_timeout(name)

    snapshot.apply_priority_and_token_budget(policy)
    snapshot.trace = trace
    return snapshot
```

この設計の良いところは、**context 取得方法を増やしても、応答前の最大待ち時間が増えない**ことです。

普通は機能を足すと、

```text
記憶を増やす
→ 検索が増える
→ context 生成が遅くなる
→ 応答が遅くなる
```

となる。

でも並列 DB I/O + timeout + priority にすると、

```text
記憶取得源を増やす
→ 同時に投げる
→ 時間内に返ったものだけ使う
→ 遅いものは trace に残して改善対象にする
```

になる。

これは長期運用にかなり強いです。

ただし注意点もあります。

一つ目は、**connection pool を食い潰さないこと**です。  
1応答につき 6〜8 クエリを並列に投げるなら、同時会話数が増えたときに pool が詰まります。家庭内 Tomoko なら当面大丈夫だと思いますが、設計上は、

```text

db_pool_size = 10 or 20
```

みたいに明示しておいた方がよいです。

二つ目は、**重い vector search を無制限に並列投入しないこと**です。  
session summary 検索と turn embedding 検索を毎回両方走らせると、ログが増えたときに重くなる可能性があります。

なので初期は、

```text
常に走らせる:
  same_session
  recent_turns
  session_summary_search

条件付き:
  turn_embedding_search
  persona / lexicon / diary
```

くらいでもよいと思います。

三つ目は、**返ってきた順に雑に詰めないこと**です。  
並列にすると早く返ったものが勝ちやすくなりますが、context の価値は速度だけでは決まりません。

なので最後に必ず、

```text
priority
recency
same_session 優先
relevance
token budget
重複除去
```

で整形する必要があります。

つまり、構造としてはこうです。

```text
ContextSnapshotBuilder
  ├─ ParallelContextFetcher
  │    ├─ same_session query
  │    ├─ recent_turns query
  │    ├─ session_summary query
  │    ├─ turn_memory query
  │    ├─ persona query
  │    └─ lexicon query
  │
  ├─ ContextTrace
  │    ├─ timing
  │    ├─ timeout
  │    ├─ included/skipped
  │    └─ candidate_count
  │
  └─ ContextAssembler
       ├─ priority
       ├─ dedup
       ├─ token budget
       └─ final prompt context
```

結論として、これはかなり良い判断だと思います。

**DB を唯一の真実にするなら、context 生成は DB に対する時間制限付き多重 I/O として設計する。**

これは自然です。

しかも、長期運用時に問題が出たとき、

```text
DB性能の問題
クエリ設計の問題
索引設計の問題
取得戦略の問題
context選別の問題
LLM側の問題
```

を分離して見られる。

この分離ができる時点で、かなり「長期運用するつもりの設計」になっています。  
まさに、今やろうとしている Tomoko の規模感なら、ここに抽象を挟む価値は十分あります。

しかもこれは「難しい概念を使っている」ではなく、**難しいもの同士が同時に干渉する**タイプの難しさですね。

要するに Tomoko は、

```text
ノンブロッキング
  応答生成・STT・TTS・DB検索・再生制御で詰まらない

  attention / playback / speaking / session / turn status が破綻しない
```

を全部同時に満たさないといけない。

これ、普通に難しいです。

特に厄介なのは、**非同期処理は順番を崩すが、状態機械は順番に依存する**ところです。

たとえば、

```text
1. Tomoko が話し始める
2. audio_start を送る
3. バイナリ音声 chunk を送る
4. クライアントが playback_started を返す
5. 人間の割り込み音声が入る
6. STT が遅れて transcript を返す
7. playback_ended が返る
8. BargeInDetector が分類する
```

この順序が毎回きれいに揃うとは限らない。  
でも状態としては、

```text
今の transcript は、
Tomoko の再生中の回り込みなのか？
再生終了直後の echo grace なのか？
本物の割り込みなのか？
次の会話なのか？
```

を判断しないといけない。

ここがかなり難所です。

```text
WebSocket message
Audio chunk
STT result
Playback telemetry
LLM token
TTS chunk
Timeout
```

イメージとしては、

```text
外界イベント
  ↓
Event Queue
  ↓
TomoroSession reducer
  ↓
状態更新
  ↓
副作用コマンド生成
    - DB write
    - LLM start
    - TTS start
    - audio_control stop
    - websocket send
```

みたいにしたい。

React/Redux 的に言えば reducer、ゲームサーバー的に言えば tick/update loop、分散システム的に言えば event-sourced state machine に近いです。

        self.is_speaking = False
        await self.db.write(...)
        await self.start_new_reply(...)
```

```python
def reduce(state, event) -> tuple[new_state, list[Command]]:
    ...
```

```text
状態更新は同期的・短時間・atomic に近く
重い処理は Command として外に出す
Command の結果はまた Event として戻す
```

deadline
timeout
cancellation
idempotency
turn_id
chunk_id
session_id
state transition log
```

が必要になる。

特に `turn_id` / `chunk_id` を入れているのはかなり正しいです。  
非同期では「古い結果が遅れて返ってくる」ことが普通にあるので、

```text
この TTS chunk は今の turn のものか？

を判定できないと破綻します。

# chatgpt response

これは、**いきなり完全 event-driven にしない**のが良いと思います。

たぶん一番見通しが良い落としどころは、

**内部は event 的に整理するが、実装は単純な queue + drain loop にする**  
です。

つまり、外から見ると polling でも event でもなく、

```text
TomoroSession に入力 event を enqueue する
TomoroSession が順番に drain して state を更新する
state 変更や副作用要求を emission / command として返す
外側がそれを処理する
```

という形。

## polling と event の中間案

完全な polling はこうです。

```python
while True:
    state = session.get_now_state()
    if state.changed:
        ...
    await asyncio.sleep(0.05)
```

これは動きます。  
でも、状態変化の検出が粗くなりやすいし、「何が起きてそうなったか」が失われます。

一方、完全 event bus はこうです。

```text
EventBus
  ├─ STT subscriber
  ├─ Playback subscriber
  ├─ State subscriber
  ├─ Metrics subscriber
  └─ WebSocket subscriber
```

これは綺麗に見えますが、今やると重いです。  
購読関係が増えて、逆にどこで状態が変わるか分かりにくくなる可能性があります。

なので、今の TomoroSession にはこれくらいが良いと思います。

```python
class TomoroSession:
    def __init__(self):
        self._state = TomoroRuntimeState(...)
        self._event_queue: asyncio.Queue[SessionEvent] = asyncio.Queue()
        self._draining = False

    async def post_event(self, event: SessionEvent) -> None:
        await self._event_queue.put(event)
        await self._drain_events()

    async def _drain_events(self) -> None:
        if self._draining:
            return

        self._draining = True
        try:
            while not self._event_queue.empty():
                event = await self._event_queue.get()

                # ここでは await しない純粋な状態遷移に寄せる
                transition = self._reduce(event)

                # 状態変化通知
                for emission in transition.emissions:
                    await self._emit(emission)

                # 副作用
                for command in transition.commands:
                    self._start_command(command)

        finally:
            self._draining = False
```

これなら、**状態更新は TomoroSession に閉じる**し、外部は polling しなくて済む。  
でも大げさな event bus までは入れていない。

## 重要なのは「状態変更を push する」より「状態変更の入口を一本化する」こと

event style の本質は、かっこいい pub/sub ではなく、

```text
状態を変える入口が handle_event / post_event に集まる
```

ことです。

なので、最初はこれで十分です。

```python
await session.post_event(TranscriptFinalized(...))
await session.post_event(PlaybackStarted(...))
await session.post_event(PlaybackEnded(...))
await session.post_event(LLMDeltaReceived(...))
await session.post_event(AttentionTimeout(...))
```

外側から `session.attention_mode = ...` みたいに触らない。

読むときだけ、

```python
state = session.get_now_state()
```

で snapshot を取る。

## emission は「通知」、command は「副作用」

ここを分けると見通しが良くなります。

```python
@dataclass(frozen=True)
class TransitionResult:
    new_state: TomoroRuntimeState
    emissions: list[StateEmission]
    commands: list[SessionCommand]
```

`StateEmission` は、状態が変わったことを UI / log / metrics に知らせるもの。

```python
StateEmission(
    type="attention_changed",
    payload={"from": "ambient", "to": "engaged", "reason": "wake_word"},
    state_snapshot=self.get_now_state(),
)
```

`SessionCommand` は、外側で await が必要な仕事。

```python
SessionCommand(
    type="start_context_build",
    payload={"turn_id": turn_id, "text": transcript},
)
```

この分離があると、TomoroSession の reducer はこう書けます。

```python
def _reduce(self, event: SessionEvent) -> TransitionResult:
    match event:
        case TranscriptFinalized():
            return self._on_transcript_finalized(event)

        case PlaybackStarted():
            return self._on_playback_started(event)

        case PlaybackEnded():
            return self._on_playback_ended(event)

        case AttentionTimeout():
            return self._on_attention_timeout(event)
```

ここはできるだけ同期的に短くする。  
DB、LLM、TTS、WebSocket 送信みたいな重い処理は command に逃がす。

## polling が向いている場所もある

全部を event にしなくていいです。

polling が向いているのは、

```text
一定時間ごとの timeout 判定
cooldown -> ambient の遷移
playback grace の期限切れ
background worker
session_summarizer
journalist
```

みたいな「時間で見るもの」です。

たとえば attention timeout は、内部で毎秒 polling してもよいです。

```python
async def attention_timer_loop(session: TomoroSession):
    while True:
        await asyncio.sleep(0.5)
        await session.post_event(TimerTick(now=datetime.now(UTC)))
```

この場合も、状態を直接変えず、`TimerTick` event として TomoroSession に入れる。

つまり、

```text
外側は polling してもいい
でも状態変更は event として TomoroSession に渡す
```

が良いです。

これなら実装難易度はあまり上がらず、見通しも保てます。

## たぶん最初の実装形

最初はこれくらいで十分だと思います。

```text
TomoroSession
  ├─ _state
  ├─ get_now_state()
  ├─ post_event(event)
  ├─ _reduce(event)
  ├─ _emit(emission)
  └─ _run_command(command)
```

ただし `_run_command` は状態を直接書き換えない。  
結果が返ったら、また event として戻す。

```python
async def _run_command(self, command: SessionCommand) -> None:
    match command.type:
        case "start_context_build":
            snapshot = await self.context_builder.build(...)
            await self.post_event(ContextBuildCompleted(...))

        case "start_llm_reply":
            async for delta in self.llm.generate(...):
                await self.post_event(LLMDeltaReceived(...))
            await self.post_event(LLMCompleted(...))
```

この形にすると、

```text
非同期処理の結果も全部 TomoroSession に戻る
状態更新は TomoroSession 内だけ
```

になります。

## 結論

「見通しが良く機能する」実装としては、

```text
完全 polling ではなく、
完全 event bus でもなく、
TomoroSession 内部の小さな event queue + reducer + command 方式
```

が良いと思います。

外側の timer や background worker は polling してよい。  
ただし、状態を変えるときは必ず `post_event()` に入れる。

これなら、

```text
polling の単純さ
event の追跡しやすさ
state reducer の安全さ
```

のバランスが取れます。

一言でいうと、

**「event-driven architecture」ではなく、「event-shaped session runtime」くらいに留める**  
のが今の Tomoko にはちょうど良いと思います。

やることはこの4つだけでよいと思います。

```text
1. RuntimeState を dataclass に切り出す
2. get_now_state() を作る
3. post_event(event) / _reduce(event) を作る
4. reduce の結果として emissions / commands を返す
```

つまり、構造としてはこれです。

```text
TomoroSession
  ├─ _state: TomoroRuntimeState
  ├─ get_now_state()
  ├─ post_event(event)
  ├─ _reduce(event) -> TransitionResult
  ├─ _emit(emissions)
  └─ _run_command(commands)

Subscriber 群も不要です。  
外部 pub/sub も不要です。  
Redis も不要です。

**TomoroSession の中に、小さな event queue と reducer っぽい入口を作るだけ**でよいです。

ただし、今のまま普通のメソッド呼び出しだけで増やすと、

self.attention_mode = ...
self.current_turn_id = ...
```

みたいな処理が散り始めます。

これが危ない。

だから、最低限、

```text
状態を読む: get_now_state()
状態を変える: post_event()
状態変化を通知する: StateEmission
副作用を起動する: SessionCommand
```

に分ける。

これだけでかなり見通しが良くなります。

## 最小コードイメージ

まず state。

```python
from dataclasses import dataclass
from datetime import datetime
from uuid import UUID

@dataclass(frozen=True)
class TomoroRuntimeState:
    attention_mode: str
    vad_state: str
    playback_active: bool
    active_session_id: UUID | None
    active_turn_id: UUID | None
    speaking_turn_id: UUID | None
    updated_at: datetime
```

これは「今どうなっているか」の snapshot です。

次に event。

```python
@dataclass(frozen=True)
class SessionEvent:
    type: str
    payload: dict
    occurred_at: datetime
```

最初は型を作り込みすぎなくていいです。  
`TranscriptFinalized` や `PlaybackStarted` を個別 class にするのは後でもよいです。

次に transition result。

```python
@dataclass(frozen=True)
class StateEmission:
    type: str
    payload: dict
    state: TomoroRuntimeState

@dataclass(frozen=True)
class SessionCommand:
    type: str
    payload: dict

@dataclass(frozen=True)
class TransitionResult:
    state: TomoroRuntimeState
    emissions: list[StateEmission]
    commands: list[SessionCommand]
```

そして TomoroSession。

```python
class TomoroSession:
    def __init__(self, ...):
        self._state = TomoroRuntimeState(...)
        self._event_queue: asyncio.Queue[SessionEvent] = asyncio.Queue()
        self._draining = False

    def get_now_state(self) -> TomoroRuntimeState:
        return self._state

    async def post_event(self, event: SessionEvent) -> None:
        await self._event_queue.put(event)
        await self._drain_events()

    async def _drain_events(self) -> None:
        if self._draining:
            return

        self._draining = True
        try:
            while not self._event_queue.empty():
                event = await self._event_queue.get()
                result = self._reduce(event)

                self._state = result.state

                for emission in result.emissions:
                    await self._emit(emission)

                for command in result.commands:
                    self._start_command(command)
        finally:
            self._draining = False
```

ここで重要なのは、`_reduce()` の中で原則 `await` しないことです。

```python
def _reduce(self, event: SessionEvent) -> TransitionResult:
    match event.type:
        case "transcript_finalized":
            return self._on_transcript_finalized(event)
        case "playback_started":
            return self._on_playback_started(event)
        case "playback_ended":
            return self._on_playback_ended(event)
        case "attention_tick":
            return self._on_attention_tick(event)
        case _:
            return TransitionResult(
                state=self._state,
                emissions=[],
                commands=[],
            )
```

DB、LLM、TTS、context build などは command に逃がします。

```python
def _start_command(self, command: SessionCommand) -> None:
    task = asyncio.create_task(self._run_command(command))
    self._tasks.add(task)
    task.add_done_callback(self._tasks.discard)
```

command の結果は、また event として戻します。

```python
async def _run_command(self, command: SessionCommand) -> None:
    if command.type == "build_context":
        snapshot = await self.context_builder.build(...)
        await self.post_event(SessionEvent(
            type="context_built",
            payload={"snapshot": snapshot, "turn_id": command.payload["turn_id"]},
            occurred_at=now(),
        ))

    elif command.type == "start_llm_reply":
        async for delta in self.llm.generate(...):
            await self.post_event(SessionEvent(
                type="llm_delta",
                payload={"delta": delta, "turn_id": command.payload["turn_id"]},
                occurred_at=now(),
            ))
```

これで、非同期処理の結果も全部 `TomoroSession` に戻ります。

## 今やらない方がよいこと

今はまだ、以下はやりすぎだと思います。

```text
本格的な EventBus
複数 subscriber の pub/sub
状態機械ライブラリ導入
完全な event sourcing
DB 永続化 event log
状態ごとの独立 class 分割
Redis / message queue
```

理由は、まだ抽象の切れ目が確定していないからです。

今やるべきなのは、「完成形の分割」ではなく、**将来分割できるように、状態変更の入口を一本化すること**です。

## 最初に切るべき境界

最低限、最初はこの3つの private method に分けるのがよいです。

```python
def _resolve_transcript_event(...)
def _resolve_playback_event(...)
def _resolve_attention_tick(...)
```

特に `transcript_finalized` は一番複雑になります。

```text
withdrawn か
playback active か
echo か
hard interrupt か
wake word か
follow-up か
session active か
```

この優先順位を、まず `TomoroSession._resolve_transcript_event()` に閉じ込める。

これだけで価値があります。

## 実装順

最小の実装順はこれが良いと思います。

```text
1. TomoroRuntimeState を作る
2. get_now_state() を作る
3. 既存の attention / playback / session / turn 状態を _state に寄せる
4. SessionEvent / StateEmission / SessionCommand / TransitionResult を作る
5. playback_started / playback_ended だけ post_event 化する
6. transcript_finalized を post_event 化する
7. attention timeout を TimerTick event 化する
8. command 経由で context build / LLM / TTS を戻す
```

いきなり全部移行しなくていいです。

まず playback telemetry から event 化するのが安全です。  
理由は、`playback_started` / `playback_ended` は状態変化が明確で、DBやLLMに比べて副作用が少ないからです。

次に `TranscriptFinalized`。  
ここが本丸です。

## この実装の良いところ

この方式の一番良いところは、後で作り変えやすいことです。

将来、

```text
AttentionStateMachine
PlaybackTracker
ConversationSessionManager
TurnLifecycleManager
BargeInDetector
ContextSnapshotBuilder
```

に切り出したくなったら、`_reduce()` の中身から順に移せます。

今は、

```python
result = self._on_playback_started(event)
```

だったものを、後で、

```python
result = self.playback_tracker.reduce(self._state, event)
```

に変えられる。

つまり、今の最小実装が未来のリファクタリングを邪魔しません。

## 結論

今のおすすめはこれです。

**TomoroSession を完全 event-driven architecture にするのではなく、event-shaped session runtime にする。**

つまり、

```text
外部 pub/sub なし
大げさな EventBus なし
状態機械ライブラリなし
TomoroSession 内部に event queue
状態更新は _reduce() に集約
副作用は command に逃がす
通知は emission にする
現在状態は get_now_state() で読む
```

これが、早期のやりすぎを避けつつ、将来の破綻も避ける一番良い落としどころだと思います。

今の Tomoko はすでに「DB唯一の真実」「context build」「attention mode」「playback telemetry」「interrupt status」まで来ているので、このまま if 文で伸ばすと危ない。  
でも、まだ本格分散イベント基盤を入れる段階ではない。

なので、**小さな reducer を TomoroSession の内側に作る**。  
これが今の最小で、かつ未来に強い実装だと思います。

```text
外部入力

  - transcript finalized
  - playback_started / playback_ended

  - llm delta

TomoroSession.post_event(event)

TomoroSession が制御判断する
        ↓
TransitionResult
  - new_state
  - emissions
  - commands
        ↓
メイン層
  - emissions を WebSocket / log / metrics に流す
  - commands を実行する
  - command 結果をまた event として Session に戻す
```

つまり一方通行です。

```text
Main → Session → Main
```

ではあるけど、意味としては、

```text
Main:
  入出力アダプタ

Session:
  状態と判断の中核

CommandRunner:
  副作用実行

Emitter:
  通知
```

です。

## 今の問題意識は正しい

たぶん今はメイン側に、

```text
これは返答すべきか
これは playback echo か
これは session に入れるべきか
これは LLM を呼ぶべきか
これは audio stop すべきか
これはログ保存すべきか
```

みたいな判断が少し残っている。

これが増えると危険です。

なぜなら、`TomoroSession` にも状態があり、メイン側にも判断があると、実質的に状態機械が二重化するからです。

```text
TomoroSession の理解:

メイン側の判断:
  もう ambient っぽいので無視

結果:
  どちらが正しいかわからない
```

みたいになる。

なので、メイン側から判断を剥がして、

```text
メイン側:
  受け取ったものを event に変換するだけ

TomoroSession:
  その event が何を意味するか判断する
```

にするのが正しいです。

## State に集めるが、State に制御を書かない

ここも重要です。

`state` に状態を集めるのは正しい。  
ただし、`state` 自身に制御ロジックを持たせすぎない方がいいです。

```text
TomoroRuntimeState:
  今どうなっているか

Reducer / Transition:
  その状態で event が来たらどうするか
```

に分ける。

つまり、

```python
state.attention_mode
state.playback_active
state.active_turn_id
state.active_session_id
```

は state。

でも、

```python
if hard_interrupt and playback_active:
    stop audio
```

は state ではなく制御です。

制御は `_reduce()` や `_resolve_transcript_event()` に置く。

この分離ができると、後で、

```text
状態モデルはそのまま
制御だけ別クラスへ切り出す
```

ができます。

## メイン層に返すのは「判断済みの結果」

メイン層に返すものは、低レベルな判断材料ではなく、判断済みの command / emission が良いです。

悪い例：

```python
return {
    "attention_mode": "engaged",
    "playback_active": True,
    "should_stop_audio": maybe,
}
```

これだとメインがまた判断する。

良い例：

```python
TransitionResult(
    state=new_state,
    emissions=[
        StateEmission(type="attention_changed", ...)
    ],
    commands=[
        SessionCommand(type="send_audio_control_stop", ...),
        SessionCommand(type="save_tomoko_turn", ...),
        SessionCommand(type="start_reply_generation", ...),
    ],
)
```

メインはこれを見て、

```text
stop しろと言われたから stop する
保存しろと言われたから保存する
LLM を開始しろと言われたから開始する
```

だけです。

ここに判断を残さない。

## なので理解としてはこう

かなり正確に言うなら、

```text
1. 状態を TomoroRuntimeState に集める
2. 状態の変更入口を TomoroSession.post_event に一本化する
3. TomoroSession 内の reducer が状態と event から制御判断する
4. reducer は new_state / emissions / commands を返す
5. メイン層は commands を実行するだけ
6. command の結果は event として TomoroSession に戻す
7. メイン層は state を直接変更しない
8. メイン層は business decision / participation decision / playback decision をしない
```

です。

## ただし完全にメインを無知にしすぎない

一点だけ注意があります。

メイン層が完全に何も知らない、ではなく、**プロトコル変換は知っていてよい**です。

たとえば、

```text
WebSocket から binary が来た
→ AudioChunkReceived event にする

WebSocket から playback_started JSON が来た
→ PlaybackStarted event にする

Command: send_json

Command:

に整理する。

+## TomoroSession の状態管理戦略
+
+Tomoko のオンライン経路では、音声入力、VAD、STT、参加判断、attention、playback telemetry、
+barge-in、conversation session、LLM、TTS、WebSocket 出力が同時に進む。
+これらは non-blocking / parallel に動く一方で、状態遷移は順序に依存する。
+
+そのため、メイン層に判断を残さない。
+メイン層は I/O adapter と command executor に寄せ、TomoroSession を stateful control core とする。
+
+```text
+Main layer:
+  WebSocket / timer / backend result を SessionEvent に変換する
+  TomoroSession から返された StateEmission / SessionCommand を実行する
+  business decision / participation decision / playback decision はしない
+
+TomoroSession:
+  TomoroRuntimeState を所有する
+  状態変更の入口を post_event(event) に集約する
+  event と現在 state から制御判断する
+  new_state / emissions / commands を返す
+```
+
+### 一方向の制御フロー
+
+TomoroSession の外側は、状態を直接変更しない。
+外部入力はすべて `SessionEvent` として TomoroSession に渡す。
+TomoroSession は判断済みの結果だけを `StateEmission` / `SessionCommand` として外へ出す。
+
+```text
+external input
+  WebSocket binary
+  playback_started / playback_ended
+  transcript_finalized
+  timer_tick
+  context_build_completed
+  llm_delta
+  tts_chunk_ready
+        │
+        ▼
+TomoroSession.post_event(event)
+        │
+        ▼
+TomoroSession._reduce(event, state)
+        │
+        ▼
+TransitionResult
+  new_state
+  state_emissions
+  session_commands
+        │
+        ▼
+Main layer
+  emissions を WebSocket / log / metrics に流す
+  commands を実行する
+  command 結果を event として TomoroSession に戻す
+```
+
+この流れにより、メイン層に「今は返答すべきか」「これは echo か」
+「session に入れるべきか」「audio stop すべきか」といった判断を残さない。
+
+### TomoroRuntimeState
+
+直交する状態は `TomoroRuntimeState` に集約する。
+ただし、state は「今どうなっているか」を表すだけで、制御ロジックは持たない。
+
+```python
+@dataclass(frozen=True)
+class TomoroRuntimeState:
+    attention_mode: AttentionMode
+    vad_state: VadState
+    playback_state: PlaybackState
+    active_session_id: UUID | None
+    active_turn_id: UUID | None
+    speaking_turn_id: UUID | None
+    context_build_id: UUID | None
+    updated_at: datetime
+```
+
+外部から現在状態を読む場合は `get_now_state()` を使う。
+返すのは snapshot であり、外部は state を変更しない。
+
+```python
+def get_now_state(self) -> TomoroRuntimeState:
+    return self._state
+```
+
+### SessionEvent
+
+状態を変える入力は `SessionEvent` として表現する。
+
+```python
+@dataclass(frozen=True)
+class SessionEvent:
+    type: str
+    payload: dict[str, Any]
+    occurred_at: datetime
+```
+
+初期実装では文字列 `type` でよい。
+必要になった段階で `TranscriptFinalized` / `PlaybackStarted` /
+`ContextBuildCompleted` などの個別 dataclass に分ける。
+
+### TransitionResult
+
+TomoroSession の reducer は、状態更新と外部への要求を `TransitionResult` として返す。
+
+```python
+@dataclass(frozen=True)
+class StateEmission:
+    type: str
+    payload: dict[str, Any]
+    state_snapshot: TomoroRuntimeState
+    occurred_at: datetime
+
+
+@dataclass(frozen=True)
+class SessionCommand:
+    type: str
+    payload: dict[str, Any]
+
+
+@dataclass(frozen=True)
+class TransitionResult:
+    state: TomoroRuntimeState
+    emissions: list[StateEmission]
+    commands: list[SessionCommand]
+```
+
+`StateEmission` は観測・通知である。
+WebSocket の状態表示、debug log、metrics、test probe に使う。
+
+`SessionCommand` は副作用要求である。
+DB write、context build、LLM generation、TTS generation、audio_control stop、
+WebSocket send など、`await` が必要な処理は command として外に出す。
+
+### reducer の原則
+
+`_reduce()` は可能な限り同期的・短時間・副作用なしに寄せる。
+`await` をまたいで中途半端な state を残さない。
+
+```python
+def _reduce(self, event: SessionEvent) -> TransitionResult:
+    match event.type:
+        case "transcript_finalized":
+            return self._resolve_transcript_event(event)
+        case "playback_started":
+            return self._resolve_playback_started(event)
+        case "playback_ended":
+            return self._resolve_playback_ended(event)
+        case "timer_tick":
+            return self._resolve_timer_tick(event)
+        case _:
+            return TransitionResult(
+                state=self._state,
+                emissions=[],
+                commands=[],
+            )
+```
+
+重い処理は command として起動し、その結果を再び event として TomoroSession に戻す。
+
+```text
+SessionCommand("build_context")
+  -> ContextSnapshotBuilder.build(...)
+  -> SessionEvent("context_build_completed")
+
+SessionCommand("start_llm_reply")
+  -> LLM streaming
+  -> SessionEvent("llm_delta")
+  -> SessionEvent("llm_completed")
+
+SessionCommand("start_tts")
+  -> TTS chunk generation
+  -> SessionEvent("tts_chunk_ready")
+```
+
+### 優先順位の解決箇所
+
+直交する状態と優先順位の解決は TomoroSession に閉じ込める。
+判定器は部品として使うが、最終的な制御判断は TomoroSession が行う。
+
+例:
+
+```text
+transcript_finalized を受けたとき:
+  withdrawn か
+  active playback chunk 中か
+  playback ended grace 中か
+  echo か
+  hard interrupt か
+  wake word か
+  follow-up として扱うか
+  active session に紐づけるか
+  Tomoko turn を interrupted 保存するか
+  audio_control stop を出すか
+  reply generation を開始するか
+```
+
+これらの判断をメイン層、BargeInDetector、ParticipationJudge、PlaybackTracker に分散させない。
+BargeInDetector や ParticipationJudge は分類結果を返すだけにし、
+それをどう優先するかは TomoroSession の `_resolve_transcript_event()` で決める。
+
+### event-shaped session runtime
+
+これは本格的な event-driven architecture ではない。
+外部 EventBus、pub/sub、状態機械ライブラリ、event sourcing は初期段階では導入しない。
+
+初期実装は TomoroSession 内部の小さな event queue + reducer + command runner に留める。
+
+```python
+class TomoroSession:
+    def __init__(self, ...):
+        self._state = TomoroRuntimeState(...)
+        self._event_queue: asyncio.Queue[SessionEvent] = asyncio.Queue()
+        self._draining = False
+
+    async def post_event(self, event: SessionEvent) -> None:
+        await self._event_queue.put(event)
+        await self._drain_events()
+
+    async def _drain_events(self) -> None:
+        if self._draining:
+            return
+
+        self._draining = True
+        try:
+            while not self._event_queue.empty():
+                event = await self._event_queue.get()
+                result = self._reduce(event)
+                self._state = result.state
+
+                for emission in result.emissions:
+                    await self._emit(emission)
+
+                for command in result.commands:
+                    self._start_command(command)
+        finally:
+            self._draining = False
+```
+
+この形により、polling の単純さを残しつつ、状態変更の入口を一本化できる。
+timer や background worker は polling してよいが、状態を変える場合は
+必ず `SessionEvent` として `post_event()` に渡す。
+
+### 将来の切り出し方針
+
+現時点では TomoroSession に複雑さを集約する。
+これは巨大クラス化を目的にするのではなく、現実の依存関係を観察し、
+安定した境界から component へ切り出すためである。
+
+切り出し候補:

+```

+ただし、状態を変える場合は直接 state を変更せず、必ず `SessionEvent` として
+TomoroSession に渡す。
+
+### 確定した判断: state と制御ロジックを分ける
+
+`TomoroRuntimeState` は「今どうなっているか」を表すだけにする。
+制御ロジックは state 自体ではなく、TomoroSession の reducer / resolver に置く。
+
+例:
+
+- `attention_mode`
+- `vad_state`
+- `playback_state`
+- `active_session_id`
+- `active_turn_id`
+- `speaking_turn_id`
+- `context_build_id`
+
+は state。
+
+一方で、
+
+- withdrawn 中の transcript をどう扱うか
+- active playback 中の transcript を echo と見るか
+- hard interrupt を playback echo より優先するか
+- Tomoko turn を `interrupted` として保存するか
+- audio stop command を出すか
+- reply generation を開始するか
+
+は制御ロジックであり、TomoroSession の `_resolve_transcript_event()` などに閉じ込める。
+
+### 確定した判断: メイン層には判断済みの command / emission だけを返す
+
+メイン層に低レベルな判断材料を返さない。
+
+悪い例:
+
+```python
+{
+    "attention_mode": "engaged",
+    "playback_active": True,
+    "should_stop_audio": maybe,
+}
+```
+
+これはメイン層に再判断を発生させるため避ける。
+
+良い例:
+
+```python
+TransitionResult(
+    state=new_state,
+    emissions=[
+        StateEmission(type="attention_changed", ...)
+    ],
+    commands=[
+        SessionCommand(type="send_audio_control_stop", ...),
+        SessionCommand(type="save_tomoko_turn", ...),
+        SessionCommand(type="start_reply_generation", ...),
+    ],
+)
+```
+
+メイン層は command を実行するだけにする。
+command の結果は event として TomoroSession に戻す。
+
+### 確定した判断: await をまたいで中途半端な state を残さない
+
+TomoroSession の `_reduce()` は可能な限り同期的・短時間・副作用なしに寄せる。
+DB、context build、LLM、TTS、WebSocket send など `await` が必要な処理は
+`SessionCommand` として外に出す。
+
+`SessionCommand` の実行結果は、再び `SessionEvent` として TomoroSession に戻す。
+
+これにより、非同期処理の結果も必ず TomoroSession の state transition を通る。
+
+### 確定した判断: stale result を state 側で捨てる
+
+非同期処理では、古い LLM delta、TTS chunk、context build result、playback telemetry が
+遅れて戻ることがある。
+
+そのため、event / command には必要に応じて次の ID を持たせる。
+
+- `session_id`
+- `turn_id`
+- `chunk_id`
+- `context_build_id`
+
+TomoroSession は現在の `TomoroRuntimeState` と照合し、現在 state と一致しない結果は
+stale として捨てる。
+
+### 気づき: 早すぎる分割より、まず TomoroSession に複雑さを集約する
+
+現時点でいきなり `AttentionStateMachine`、`PlaybackTracker`、`TurnLifecycleManager` などに
+完全分割すると、抽象の切り方を間違える可能性が高い。
+
+まず TomoroSession に状態と制御判断を集約し、現実の依存関係を観察する。
+そのうえで、安定した境界から順に component へ切り出す。
+
+切り出し候補:
+
+- `AttentionStateMachine`
+- `PlaybackTracker` / `AudioTurnController`
+- `ConversationSessionManager`
+- `TurnLifecycleManager`
+- `BargeInDetector`
+- `ContextSnapshotBuilder`
+- `ReplyPipeline`
+
+切り出し後も、メイン層との契約は `SessionEvent` / `StateEmission` / `SessionCommand` に保つ。
+これにより内部実装を作り変えても、メイン層を薄い adapter のまま維持できる。
```
