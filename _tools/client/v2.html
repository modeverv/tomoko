🎙️ Tomoko — Turn-taking v2 シグナル/データフロー

対象: make server-debug（配信プロセス）+ make turn-taking-v2-worker（シャドウワーカー）。 プロセス間のシグナルは PostgreSQL の pg_notify 2チャネルのみ。データは常に一方向に流れ、逆流はない。

1. プロセス間データフロー（全体像）

音声・VAD/VAP 信号 STT テキスト DB 書き込み + pg_notify シャドウ評価・advisory メインレーン（応答生成）

2. 1ターンのタイミングモデル（ベンチ TT-v2.9 で計測済み）

fusion は log-only で並走中。実機ログ突合（TT-v2.10d）でベンチ予測と一致を確認後、仮推論のトリガーを fusion に切り替える。 prefill API（TT-v2.10c）はバックエンドプリミティブ実装済み・結線はトリガー切替と同時。

3. データの着地点（誰が書き、誰が読むか）

場所書き手読み手主な内容
partial_transcript_observations session（_maybe_emit_partial_transcript） shadow worker（LISTEN/リカバリ） raw_text, vad_state, audio_level_db, p_yielding（TT-v2.10a）
turn_taking_v2_advisories shadow worker（save_advisory） session（advisory リスナー） saturation, proposal, would_start_inference, fusion_score / would_start_inference_fusion（TT-v2.10b）
logs/turn-taking-main.jsonl session（メインレーン・仮推論イベント） analyze_turn_taking_v2.py final transcript の決定、provisional_inference の start/discard
logs/turn-taking-v2.jsonl shadow worker（log_v2_shadow_advisory） analyze_turn_taking_v2.py（fusion 比較付き） advisory 全件（p_yielding, fusion_score 含む）
logs/server-debug.log make server-debug（tee） 人間 uvicorn / DEBUG ログ全文
logs/turn-taking-v2-worker.log make turn-taking-v2-worker 人間 worker の処理ログ（LLM判定の生応答含む）

4. 設計原則（このフローが守っている規約）

  • 一方向フロー: gateway → observations → worker → advisories → session。逆向きの書き込みは存在しない。
  • プロセス間シグナルは pg_notify 2チャネルのみ: turn_taking_v2_observation と turn_taking_v2_advisory。payload は UUID のみで、データ本体は必ずテーブルから読む。
  • shadow は助言のみ: worker はメインレーンの動作を直接変更しない。session 側が advisory を読んで仮推論を起動/破棄する。
  • log-only で検証してから切替: fusion 判定（TT-v2.10b）は記録のみ。実機ログとベンチ予測の突合（TT-v2.10d）後にトリガーを切り替える。
  • 状態はコンポーネント内に閉じる: KVキャッシュ（prefill、TT-v2.10c）は MLXLMBackend 内のみ。ターン内 append-only・ターン終了で破棄。
